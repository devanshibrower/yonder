Moon iterations — what to keep from each

Comparing the inline src/app/experiments/blog-moon/page.tsx against the canonical src/components/ui/MoonCanvas.tsx (rendered at / and /experiments/moon). Verdicts reflect decisions made.

▶ MoonCanvas / ▶ MoonCanvas /experiments/moon ▶ blog/moon /experiments/blog-moon ▶ All experiments /experiments

01 Idle render loop keep from blog/moon

blog/moon stops asking for animation frames once motion settles. MoonCanvas redraws all 365 ticks + moon + labels at 60fps forever, even when nothing moves.

blog/moon ✓page.tsx:300
// re-arm only while something animates
const isAnimating =
  Math.abs(arcDiff) > 0.5 || Math.abs(phaseDiff) > 0.005;
if (isAnimating) {
  rafId = requestAnimationFrame(draw);
}
MoonCanvas ✗MoonCanvas.tsx:447
// runs every frame, no exit condition


  rafRef.current = requestAnimationFrame(draw);
// ...60fps for the life of the page

Why it matters: constant canvas clearing + a 365-iteration loop pins a CPU core and drains battery on a static screen. The idle version wakes only on hover/select.

02 Ring-band hit detection keep from blog/moon

blog/moon only treats the mouse as "on the dial" when it's inside the ring's band. MoonCanvas converts any point on the 360×360 canvas to a day — so hovering the moon in the center, or the empty corners, still scrubs the phase.

blog/moon ✓page.tsx:343
const dist = Math.sqrt(x*x + y*y);
if (dist < RING_RADIUS - 14 ||
    dist > RING_RADIUS + 14) {
  setHoveredDay(null); return;
}
MoonCanvas ✗MoonCanvas.tsx:277
// no distance check at all
let mouseAngle = Math.atan2(mouseY, mouseX);
// any pixel → a day
setHoveredDay(day);

Why it matters: hovering the moon graphic itself shouldn't yank the date around. The band makes the dial feel like a dial.

03 Arc direction at the year boundary keep MoonCanvas's behavior

The arc is a progress fill anchored at Jan 1. On a big jump (e.g. Dec → Jan) the two iterations differ. Decision: follow the day numbers. Going from day 354 to day 5 is moving earlier in the year, so progress should rewind 97% → 1.4% — exactly what MoonCanvas already does with a plain lerp. blog/moon takes a spatial shortcut across the top, which misrepresents a single-year dial.

MoonCanvas ✓MoonCanvas.tsx:350
// straight lerp — follows day number
animatedArcRef.current +=
  (arcTarget - animatedArcRef.current)
  * 0.12;
blog/moon ✗page.tsx:195
let arcDiff = selectedDay - animatedArc;
if (arcDiff > 182.5)  arcDiff -= 365;
if (arcDiff < -182.5) arcDiff += 365;
animatedArc += arcDiff * 0.12;

Note: the moon phase lerp should still wrap (the lunar cycle is genuinely cyclic). Only the year-progress arc is linear — so phase wrapping + arc not wrapping is correct, not a conflict.

04 Progress arc lines up with the ticks keep from blog/moon

Ticks sit at angle (day − 1) / 365. blog/moon ends the arc with the matching (arc − 1) offset so the arc head lands exactly on the selected tick. MoonCanvas uses arc / 365, so the rounded cap overshoots by one day's width.

blog/moon ✓page.tsx:239
endAngle = -PI/2 +
  ((animatedArc - 1) / 365) * 2*PI;
// matches the marker formula
MoonCanvas ✗MoonCanvas.tsx:387
endAngle = -PI/2 +
  (animatedArc / 365) * 2*PI;
// drifts 1 tick past the day

Why it matters: select Jan 1 → blog/moon shows no arc (0 progress, lands on the tick); MoonCanvas shows a one-day sliver poking past it. Minor, but blog/moon is the correct fencepost.
